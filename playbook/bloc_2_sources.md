## Source 1 — Notion "Doc Process & suivi CRO" (sections persuasion)

Extraction exhaustive via `notion-fetch` sur `https://www.notion.so/growth-society/Doc-Process-suivi-CRO-Mission-Media-buying-2e67148e95a5805b8fafe8531ad5159e`.

### 1a. Section Bénéfices 🟢 (socle rationnel avant persuasion)

**Règles gravées par Growth Society :**
- "Pourquoi je devrais acheter ce produit plutôt qu'un autre… ou rien du tout ?" → l'USP doit répondre instantanément.
- **Traduire chaque feature en impact concret utilisateur**. Mauvais : "Contient 300mg d'électrolytes". Bon : "Hydratation optimale pendant l'effort".
- **Max 3–5 USP majeures**. Au-delà = surcharge cognitive, perte de hiérarchie, dilution du message.
- Format performant : titre bénéfice court + courte explication + icône/visuel.
- Critères qualité USP : Unique / Spécifique / Crédible / Important pour l'utilisateur.
- Lecture en 5 secondes obligatoire (scan possible).
- **Position** : juste après hero ou dans hero étendu, AVANT preuves sociales. "C'est le socle rationnel avant persuasion."
- **Adaptation niveau de conscience** : marché froid = USP résultat / compétitif = USP différenciante / premium = USP expérience.
- **Erreurs tuables** : trop d'USP, USP vagues ("qualité premium"), USP techniques incompréhensibles, USP identiques aux concurrents, mélange bénéfices+features sans hiérarchie.

### 1b. Section Preuves sociales / Avis clients 🟢

**Stats citées :**
- 61 % des clients lisent les avis en ligne avant d'acheter.
- Probabilité d'achat × 2,7 si ≥ 5 avis affichés.

**3 leviers activés :** validation sociale / réassurance / projection utilisateur.

**8 règles opérationnelles :**
1. **Crédibilité > quantité**. Un bon avis = prénom + photo + ville + détail concret + ressenti + bénéfice précis. Mauvais : "Super produit". Bon : "Plus de problème de récupération après mes entraînements, je l'utilise tous les matins."
2. **Avis ciblant les objections clés** : efficacité, délai, comparaison ancienne solution, facilité, qualité, rapport qualité/prix. → "Un bon avis = mini-argument de vente."
3. **Mixer formats** : note globale + avis courts + 1-2 avis détaillés + vidéo.
4. **Photos réelles > stock**.
5. **Preuve sociale placée tôt et au contact du CTA**.
6. **Hiérarchie d'efficacité** (de la plus forte à la plus faible) :
   - Vidéo témoignage ★★★★★
   - Avis vérifiés (Trustpilot, Google) ★★★★★
   - Screenshots avis ★★★★
   - Chiffres/stats ★★★★
   - Logos clients ★★★
   - Badges certifs ★★
   - Testimonial anonyme ★

### 1c. Section FAQ 🟢 — "arme de conversion anti-doute"

**Principe central Growth Society :** *"Une FAQ performante n'est pas une base d'aide. C'est une arme de conversion anti-doute."*

**Règle simple :** "Si la réponse n'influence pas l'achat → elle ne doit pas être là."

**8 checkpoints :**
1. Uniquement questions bloquantes conversion : prix/paiement, livraison, retours/garantie, efficacité, compatibilité, délais résultats, engagement abonnement.
2. **5–8 questions max**. Sweet spot CRO : **6–7 questions**. <5 = manque réassurance. >10 = surcharge cognitive.
3. **Ordre stratégique** : financier → logistique → efficacité → compatibilité/usage → support.
4. Réponses courtes (<50 mots) : réponse directe + précision rassurante + micropreuve.
5. Micro CTA dans certaines réponses (garantie, livraison gratuite).
6. Position : après buybox/bundles, après preuves sociales, avant footer. "Filet de sécurité final."
7. Accordion toggle, fermé par défaut.
8. **Source des bonnes questions** : tickets support, commentaires ads, objections sales, reviews, questions Amazon. **"Pas d'un brainstorming interne !"**

### 1d. Section Reason Why / Comparaison / Avant-Après / Personae (boosters)

- **Reason Why** : explique le mécanisme derrière les bénéfices → crédibilité rationnelle. Structure : Promesse → bénéfices → **reason why** → preuves → achat.
- **Comparaison Eux vs Nous** : assumer la comparaison, souvent plus efficace vs "vraie alternative utilisateur" (statu quo, solution actuelle) que vs concurrent direct.
- **Before/After** : preuve implicite. "Si on voit la différence, on croit plus vite."
- **Personae dans le copy** : transformer "est-ce pour moi ?" en "c'est exactement pour moi". Identification / qualification / projection.
- **Architecture cible** : Hero → Pour qui → Bénéfices → Reason Why → Preuves → FAQ → CTA.

### 1e. Cohérence copy / scent trail persuasif

- "Copywriting, design ou promesses non alignées = confusion = perte du visiteur."
- "S'assurer qu'on retrouve bien le copywriting mis en avant dans les ads dans la lp — Promesses / Pain / Bénéfices."
- Ton of voice ads → LP : si ads punchy, LP punchy. Sinon rupture de scent trail.

### 4b. Benefit Laddering (section 2.9) — structurant pour per_01

Escalade : **Feature (QUOI) → Avantage (COMMENT) → Bénéfice (POURQUOI) → Transformation (QUI SERAS-TU)**.

Exemple :
- Feature : "30 minuteurs prédéfinis"
- Avantage : "Pas besoin de configurer"
- Bénéfice : "Gagne 5 minutes chaque jour"
- Transformation : "Deviens quelqu'un de productif et organisé"

**Test "Et alors ?"** : pour chaque ligne de copy, demander "et alors ?" jusqu'à atteindre un bénéfice émotionnel. Si la ligne ne survit pas au test → c'est une feature qui n'a pas été traduite.

### 4c. Niveaux de Conscience de Schwartz (section 2.10) — structurant pour per_02 / per_07

| Niveau | Copy strategy |
|---|---|
| Unaware | Créer la conscience du problème ("Tu ignores peut-être que…") |
| Problem Aware | Montrer que des solutions existent ("Heureusement, il existe…") |
| Solution Aware | Comparer les solutions ("Comparé aux alternatives…") |
| Product Aware | Convaincre que c'est le meilleur ("Voici pourquoi on est meilleur") |
| Most Aware | Promo / urgence / objections ("Offre limitée, clique ici") |

→ Le ton et l'angle narratif doivent matcher le niveau de conscience du trafic entrant (cold vs warm vs retarget).

### 4d. Friction Cognitive (section 2.5.1) — structurant pour per_08

Signes d'alerte jargon / generic AI tone :
- Headline > 15 mots
- Jargon technique non-expliqué
- Multiples value propositions (au lieu d'une)
- "Phrases marketing génériques en volume" (cf. baseline Japhy)

| Critère | checkMethod |
