# Task: Code doctrine, hygiene linter & auto-update loop

## Description

Final task — the **anti-regression contract**. After 10 tasks of cleaning, encode the rules so future Claude sessions don't drift back into the V21→V26 brownfield. Three deliverables:

1. **`docs/doctrine/CODE_DOCTRINE.md`** — the rules of the road, with examples.
2. **`scripts/lint_code_hygiene.py`** — automated enforcement of the mechanical rules.
3. **CLAUDE.md / AGENTS.md / sub-agents patches** — the doctrine becomes part of every Claude instance's init checklist and every sub-agent's refusal criteria.

The cleanup epic is not closed until any future agent reading `CLAUDE.md` is force-fed the doctrine on session #1.

## Acceptance Criteria

### Doctrine document

- [ ] `docs/doctrine/CODE_DOCTRINE.md` exists, includes:
  - **Principle**: 1 fichier = 1 concern. The 300-LOC "threshold" is a *signal*, not a rule. The 800-LOC threshold is a *hard fail*.
  - **The 8 canonical concern axes**: prompt assembly · API client · persistence · orchestration · CLI · config · validation · I/O serialization. Every active source file must self-classify into exactly one.
  - **Rules — hard (mechanical, linter-enforced)**:
    1. No file >800 LOC in active paths.
    2. Env reads (`os.environ`, `os.getenv`) only inside `growthcro/config.py`.
    3. No `_archive*` / `_obsolete*` / `*deprecated*` / `*backup*` directory inside active paths.
    4. No basename duplicates in active paths (excluding `__init__.py`, `cli.py`).
  - **Rules — soft (heuristic, linter-warned, human/LLM-judged)**:
    1. Files >300 LOC are surfaced as `info` — reviewer affirms "single concern".
    2. Files >300 LOC with mixed-concern signals (cross-domain imports, divergent function-name prefixes, multiple unrelated top-level classes) are surfaced as `warning`.
  - **Auto-update loop spec**: when an agent discovers a new anti-pattern not yet codified, it produces a separate commit `docs(doctrine): code +<rule>` with: (a) one-line rule, (b) concrete code example of the violation it would have caught, (c) whether it should be added to the linter's `fail` set (mechanical), `warning` set (heuristic), or doctrine-text only (judgment-based).
  - **Anchor in CLAUDE.md**: the doctrine is mandatory init reading (step #10).
- [ ] Doctrine is short — target ≤200 lines, dense and example-driven.

### Linter

- [ ] `scripts/lint_code_hygiene.py` exists, runs in <5s on the cleaned tree, exits 0/1/2:
  - `0` — green
  - `1` — at least one `fail` rule violated
  - `2` — internal error (file walk failed, etc.)
- [ ] Output format:
  ```
  CODE HYGIENE — 2026-05-08
  ────────────────────────
  FAIL  3 issues
    growthcro/api/server.py: 821 LOC (limit 800)
    cli/old_thing.py: os.environ outside config.py
    skills/X/_archive_old/: archive in active path
  WARN  1 issue
    growthcro/recos/prompts.py: 412 LOC + mixed-concern signal (imports requests + jinja2)
  INFO  4 files >300 LOC (single-concern affirmed by name)
  ```
- [ ] Linter is plain Python stdlib + `pathlib` + `re`. No new dependencies.
- [ ] Wire into `state.py`: the project-state dump's final section calls the linter and prints the summary line (`CODE HYGIENE — fail: N, warn: M, info: K`).
- [ ] Running on the post-Task-10 tree exits 0 — the cleanup itself is fully compliant.

## Technical Details

- The linter's "mixed-concern signal" heuristic is intentionally simple — false positives are OK because the level is `warning`, not `fail`. Initial signals to detect:
  - Imports from ≥3 of: `{requests, httpx, urllib}`, `{sqlite3, sqlalchemy, json+pathlib}`, `{jinja2, markdown}`, `{argparse, click}`, `{anthropic, openai}`, `{playwright, selenium}`
  - Function-name prefix entropy: extract prefixes (`assemble_*`, `save_*`, `run_*`, `parse_*`); if ≥3 distinct prefixes each accounting for ≥20% of functions → warn
  - ≥2 top-level classes that don't reference each other
- The auto-update loop is **doctrine, not infrastructure**. We don't build a tool that auto-edits the doctrine — we build the social/process contract: any agent proposing a new rule must do it via a clearly-named commit. The linter's rule registry is plain Python data structures; adding a rule = editing a list and shipping a commit, period.
- This task's commits should be small and themed — the doctrine commit, the linter commit, the CLAUDE.md commit, the agents commit, the example loop commit. Five small commits beat one giant.
